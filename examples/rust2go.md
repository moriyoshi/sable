# Example: a rust2go-style typed async call, on Sable

[rust2go](https://github.com/ihciah/rust2go) is the closest prior art to Sable:
it lets **Go call async Rust** efficiently (an eventfd ring + a hand-written
g0-stack asm callback, so the awaiting goroutine doesn't pin an OS thread). You
declare a trait, its macro/CLI generates the C-ABI glue and a typed Go binding,
and calling it runs the async Rust and resumes the goroutine on completion.

This example reproduces rust2go's canonical `DemoCall.demo_check` use case on
Sable, to show the calling convention is the same — and to make the one
architectural difference concrete.

**There are two versions in this repo:**

1. **Illustrative** (`examples/rust2go/main.go`, always built, zero deps) — the rust2go
   *calling pattern* hand-written over Sable's generic `Call`. Read this first for
   the shape.
2. **The real rust2go crate, actually wired in** (opt-in: `--features rust2go` /
   `-tags sable_rust2go`) — `rust/src/r2g.rs` (the `#[rust2go::g2r]` trait),
   `examples/rust2go-real/r2g_gen.go` (**generated by `rust2go-cli`**), `examples/rust2go-real/real.go` (the glue). This
   is documented in "Actually wiring the crate" below.

## rust2go's shape

```rust
// Rust: the interface; the macro generates the FFI + a Go binding.
#[rust2go::r2g]
pub trait DemoCall {
    fn check_stock(req: &Order) -> impl Future<Output = StockReply>;
}
```

```go
// Go: call it like a local async function; the goroutine parks, no thread blocked.
reply := DemoCall.check_stock(&Order{Item: "widgets", Qty: 5})
```

Under the hood rust2go spawns the future on **its own** runtime (monoio/tokio,
with its own reactor) and calls back into Go when it resolves.

## The same thing on Sable

The Rust handler is one op in Sable's dispatch registry — the body you'd write
inside the `#[rust2go::r2g]` method (`rust/src/demo.rs`, `check_stock`, op 10):

```rust
async fn check_stock(req: Vec<u8>) -> Result<Vec<u8>, Vec<u8>> {
    let (qty, item) = decode_order(&req)?;
    tokio::time::sleep(Duration::from_millis(1)).await; // async inventory "lookup"
    let ok = qty <= 20; // 20 units on hand
    Ok(encode_reply(ok, /* message */))
}
```

The typed Go binding — the part rust2go's codegen emits, here written by hand
over Sable's generic `Call` (`examples/rust2go/main.go`):

```go
func CheckStock(ctx context.Context, o Order) (StockReply, error) {
    respBytes, err := sable.CallCtx(ctx, opCheckStock, encodeOrder(o))
    if err != nil {
        return StockReply{}, err
    }
    return decodeReply(respBytes)
}
```

Calling it is identical in spirit — the goroutine parks (via `gopark`, or a
channel in the portable backend) while the Rust future runs, and is resumed by
the dispatcher when the result lands:

```go
reply, err := CheckStock(context.Background(), Order{Item: "widgets", Qty: 5})
// reply.OK == true, reply.Message == "widgets reserved"
```

## Run it

```sh
make rust                          # build libsable.a once
go run ./examples/rust2go          # prints the rust2go-style CheckStock results
go test -race ./examples/rust2go/  # typed, concurrent, cancellable
```

`TestRust2GoStyle` fans out 500 goroutines each awaiting an async `CheckStock`
with none pinning an OS thread; `TestRust2GoStyleCancel` shows cancelling the
`context` aborts the in-flight Rust task — rust2go-parity features on the fused
runtime.

## The difference that matters

Both keep the awaiting goroutine off an OS thread; the divergence is underneath.

| | rust2go | Sable |
|---|---|---|
| Rust runtime | its **own** (monoio/tokio) with its **own reactor** | tokio with **IO disabled**, fused into Go's netpoll |
| Event loops in the process | **two** (Go's + Rust's) | **one** shared epoll (Go's netpoll) |
| Rust→Go resume | eventfd ring + g0-stack asm callback | completion doorbell + dispatcher `goready` (asmcgocall on the hot fd path) |
| Typed binding | **generated** by macro/CLI | hand-written over the byte-buffer `Call` (or your own codegen) |
| Marshalling | flat-struct codec (generated) | caller-chosen (here a flat binary codec, same idea) |

So rust2go is *fast IPC between two schedulers*; Sable is *one fused scheduler*.
For the common "Go awaits async Rust" call they look the same at the call site
(this example); Sable additionally makes I/O readiness flow through a single
epoll, which is what removes the second reactor.

## Actually wiring the crate (opt-in)

The second version uses the **real rust2go crate**, not an imitation. The key
fact that makes the fusion natural: rust2go's `g2r` (Go-calls-Rust) is
**synchronous** — its codegen literally errors on an `async fn` with *"async is
not supported yet when go call rust, manually spawn by your own!"* That "manually
spawn by your own" is exactly Sable's role.

So the division of labor is:

- **rust2go** owns the **typed request crossing**: `#[derive(R2G)]` marshals the
  struct, `#[rust2go::g2r]` generates the C export + the Go binding, and the call
  crosses on rust2go's own g0-stack `cgocall` shim. (`#[cgo]` picks the safe shim
  because the handler allocates.)
- **Sable** owns the **async execution + completion**: the g2r handler
  (`rust/src/r2g.rs`) decodes the request, then `spawn`s onto Sable's IO-disabled
  tokio runtime (fused into Go's netpoll) and delivers the result through Sable's
  completion doorbell on the await `token`. The Go caller parks via the library's
  public `sable.AwaitCallResult` primitive and is resumed by Sable's dispatcher.

```
 Go goroutine
   │  CheckStockReal(item, qty)                    examples/rust2go-real/real.go
   │  sable.AwaitCallResult(spawn):
   ▼    DemoCallImpl{}.demo_check_spawn(&order,&rt,&token)  examples/rust2go-real/r2g_gen.go (generated)
 rust2go g2r cgocall crossing  ── c_DemoCall_demo_check_spawn ──►  Rust handler
                                                                   rust/src/r2g.rs
   handler: decode Order (rust2go marshalling)
            rt.handle.spawn(async { sleep; encode; inner.complete(token, resp) })
   │                                            (Sable's FUSED runtime)
   ▼  completion doorbell → dispatcher → goready
 Go goroutine resumes with the typed StockReply
```

The response comes back through Sable's byte-buffer completion (a boxed
`CallResult` Go copies out and frees) — rust2go's g2r return is synchronous, so an
async result can't ride it; Sable's async completion path carries it instead.

### Build / run / test

```sh
# rust2go passes Go pointers through its ref structs, so it REQUIRES relaxed cgo
# checks (rust2go's own README/CI do the same). The Makefile targets set them.
make run-rust2go     # builds --features rust2go + -tags sable_rust2go, runs the demo
make test-rust2go    # typed + 200 concurrent async calls through the real binding
make gen-rust2go     # regenerate examples/rust2go-real/r2g_gen.go from rust/src/r2g.rs (needs rust2go-cli)
```

The generated `examples/rust2go-real/r2g_gen.go` is **committed**, so building/testing needs only the
`rust2go` crate + the `github.com/ihciah/rust2go` Go module (both fetched
automatically); `rust2go-cli` is needed only to *regenerate*. Everything is behind
the feature/tag and a separate cargo target dir, so the default, portable, and
certified-core builds are completely untouched.
